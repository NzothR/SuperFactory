# GTNH 工序机器详细设计文档 v2

## 1. 文档目的

本文档是 `GTNH工序机器详细设计文档.md` 的第二版设计草案，用于描述集成工厂在新版工序图模型下的机器运行模型。

第二版重点统一以下内容：

```text
目标产物节点不再等同于拓扑终点
运行器按物料边进行流水线调度
支持多源头、多目标、多副产链和多个局部单物料自增环
图语义与运行状态分离
提交新图时可继承主机和 NC
螺丝刀输出仍为强制清空
排空模式只作为边界设计记录，暂不作为当前必做行为
```

本文档只讨论机器运行模型，不重新定义配平算法。工序图结构、目标产物推断和环配平规则见 `GTNH工序图模型与速率配平算法设计_v2.md`。

---

## 2. 机器定位

集成工厂是一台：

```text
基于工序图的虚拟复合加工机
```

它不会真实启动每个节点对应的 GT 多方块主机，而是通过提交的工序图生成内部虚拟运行任务。

机器职责：

```text
接收并校验工序图
统计并收集运行资格物
维护运行期资源快照
按图结构调度虚拟节点
模拟耗时、耗能、输入消耗和输出产出
维护内部缓存、循环水位和外部输出缓存
处理换图、输出、跳电和强制清空
```

---

## 3. 图语义与运行状态分离

v2 明确区分：

```text
图语义 = 工序图定义了什么
运行状态 = 当前机器执行到了哪里、缓存了什么
```

图语义包括：

```text
节点配方
物料边
目标产物节点
主机需求
NC 需求
启动物料需求
环路规则
配平结果
```

运行状态包括：

```text
运行中的 job
内部中间物缓存
循环物料水位
目标/副产输出缓存
已收集的主机和 NC
已收集的启动物料
```

提交新图时，只允许比较和继承启动条件：

```text
主机方块
NC 物品
```

以下运行状态不参与新图语义比较：

```text
旧图中间物
旧图终产物
旧图副产物
旧图外部原料缓存
正在运行 job 已消耗的材料
```

---

## 4. 节点运行模型

每个节点运行时使用有效参数：

```text
effectiveParallelLimit = node.parallelLimit × machine.parallelMultiplier
effectiveOc = node.overclockCount + machine.extraOverclock
effectiveOc = min(effectiveOc, baseDuration 压到 1 tick 所需超频次数)
effectiveDuration = baseDuration 按 effectiveOc 压缩
effectiveEuPerTick = baseEuPerTick 按 effectiveOc 放大
```

单次 job 启动时：

```text
actualParallel <= effectiveParallelLimit
消耗 input × actualParallel
运行 effectiveDuration tick
消耗 effectiveEuPerTick × actualParallel
完成后产出 output × actualParallel
```

每个节点同一时间只允许一个 running job。

该规则保留节点作为一个独立虚拟加工单元的语义，避免同一节点无限叠加 job 导致运行模型失真。

---

## 5. 主机与 NC

### 5.1 主机

主机方块代表节点运行资格。

规则：

```text
每个节点独占一台可运行其配方类型的主机
主机不在运行期真实执行配方
主机只作为运行资格缓存
```

### 5.2 NC 物品

NC 是 Non-Consumable，表示不消耗运行资格物。

典型 NC：

```text
编程电路
模具
透镜
催化剂
特殊工具
配置物品
```

NC 不参与普通输入消耗，也不进入内部物料缓存。

### 5.3 生命周期

主机和 NC 的生命周期：

```text
INPUT：从输入舱室收集
RUNNING：保存在运行资格缓存中
OUTPUT：返还到输出舱室
提交新图：可被新图继承
螺丝刀输出：强制返还
```

---

## 6. 机器状态

机器状态建议维持：

```text
STANDBY
INPUT
RUNNING
OUTPUT
```

其中：

```text
STANDBY = 无已安装工序或等待操作
INPUT = 收集当前工序所需主机、NC 和启动物料
RUNNING = 按工序图执行虚拟节点
OUTPUT = 清空运行状态并输出缓存物料
```

排空模式 `DRAINING` 暂不作为当前实现要求，只作为未来扩展边界记录。

---

## 7. 提交工序

### 7.1 提交目标

提交新工序时，机器应尽量减少玩家重复取放主机和 NC 的操作。

### 7.2 继承规则

提交新图时，对旧图已收集的启动条件和新图需求进行比较。

可继承内容：

```text
主机方块
NC 物品
```

不可继承内容：

```text
启动消耗物料
内部中间物
目标产物缓存
副产物缓存
运行中的 job
旧图外部原料缓存
```

### 7.3 主机继承

主机继承按节点运行资格匹配。

对于新旧需求：

```text
如果旧主机能够满足新图某个主机需求，则保留
保留数量不超过新图需求数量
多余旧主机进入输出流程
未满足的新主机需求进入 INPUT 收集流程
```

### 7.4 NC 继承

NC 按物品身份匹配。

规则：

```text
新图同样需要的 NC 保留
旧图多余 NC 进入输出流程
新图仍缺 NC 进入 INPUT 收集流程
```

### 7.5 提交后的状态

如果新图需求已被继承资源完全满足：

```text
直接进入 RUNNING
```

如果仍缺主机、NC 或启动物料：

```text
进入 INPUT
```

如果旧图有不可继承的缓存或运行状态：

```text
先进入 OUTPUT，输出不可继承内容
输出完成后再安装新图或继续 INPUT
```

---

## 8. 螺丝刀输出

螺丝刀切换到输出模式是强制清空语义。

它不继承任何旧图资源。

应输出：

```text
主机方块
NC 物品
启动物料缓存
内部中间物
目标产物缓存
副产物缓存
```

正在运行 job 的处理保持当前强制输出语义：

```text
终止运行任务
不继续启动新节点
按既定规则处理已消耗/已缓存物料
```

螺丝刀输出用于玩家明确要求拆机、换线或强制回收状态。

---

## 9. 运行期资源快照

RUNNING 每 tick 应构建或维护资源快照。

资源来源：

```text
内部缓存
循环水位缓存
外部输入舱室
ME 输入舱室
双输入舱室
```

资源快照用途：

```text
计算节点可运行并行
避免同 tick 反复扫描 ME 输入
避免同 tick 后续节点看到过期资源
降低矿典匹配和输入扣取开销
```

真实输入消耗必须在 job 启动前完成。若 ME 或输入舱室物料不足，节点不启动。

---

## 10. 流水线调度模型

### 10.1 调度目标

调度器目标是让图像生产线一样连续推进，而不是下游和上游分批波动。

推荐调度策略：

```text
单 pass 调度
按边关系上游优先
目标产物节点不参与排序特殊处理
保留每节点一个 running job
保留内部/外部水位保护
```

### 10.2 上游优先顺序

调度顺序由物料边决定：

```text
无入边节点深度 = 0
节点深度 = 所有前置节点深度最大值 + 1
按深度从小到大调度
同深度按 nodeId 稳定排序
```

这允许：

```text
多个源头节点并列启动
多个目标产物节点并列输出
目标产物节点后继续连接副产处理链
```

### 10.3 环兼容

对包含环的图：

```text
先识别 SCC
DAG 部分正常上游优先
环 SCC 作为局部组参与排序
环内节点使用稳定顺序
不使用目标产物节点推断拓扑终点
```

环是否有效由图检查负责：

```text
每个环最多一种循环物料
循环物料必须正净输出
```

运行器只按检查后的图执行，不在运行期求解复杂多物料环。

---

## 11. 节点启动规则

节点可以启动需要满足：

```text
节点锁定且配方有效
无同节点 running job
有效并行上限 > 0
输入资源满足至少一次配方
输出水位未阻塞
能量可满足
```

节点实际并行：

```text
actualParallel = min(effectiveParallelLimit, 可用输入支持的最大并行)
```

如果输入不足：

```text
节点本 tick 不启动
机器继续尝试其他节点
```

---

## 12. 输入消耗规则

节点启动前按顺序消耗：

```text
1. 内部缓存中可消耗物料
2. 外部输入舱室 / ME 输入舱室
3. 双输入舱室
```

消耗必须是启动 job 的前置条件。

如果真实扣取失败：

```text
回滚本次已暂存内部消耗
节点不启动
不生成 job
```

对于 ME 输入，应尽量使用大块扣取，单次上限不超过 `Integer.MAX_VALUE`。

---

## 13. 产物路由规则

job 完成后，产物按以下顺序路由：

```text
1. 如果被直接下游消费，进入内部缓存
2. 如果属于环循环物料，先满足环 reserve
3. 如果超过环水位且被环外下游消费，作为内部可用净输出
4. 如果是目标产物，进入目标输出缓存
5. 如果无人消费且非目标，进入副产输出缓存
```

目标产物节点后的副产处理链由边决定，不由节点标记强行截断。

---

## 14. 环运行水位

每个有效环只有一种循环物料。

运行期应为该循环物料维护：

```text
reserve
lowWater
highWater
```

含义：

```text
reserve = 维持环运行不可外排的最低保留量
lowWater = 允许上游恢复生产的低水位
highWater = 暂停继续生产该循环物料的高水位
```

规则：

```text
低于 reserve 的循环物料不能被环外下游或外部输出消耗
超过 reserve 的净输出可以参与环外下游运行
超过 highWater 时暂停继续生产该循环物料的相关上游节点
```

---

## 15. 普通内部水位

普通中间物也需要水位保护，防止上游远快于下游时内部缓存暴涨。

推荐：

```text
lowWater = 下游一次有效运行所需量
highWater = lowWater 的若干倍
```

水位倍率可以配置化。默认值应偏向稳定流水线，而不是过于保守地频繁暂停上游。

---

## 16. 外部输出水位

目标产物和副产物进入输出缓存。

如果输出舱室或 ME 输出无法接收：

```text
输出缓存暂存
达到高水位后阻止继续产生该外部输出的节点启动
输出空间恢复后继续清空缓存
```

这样可以避免输出空间不足时直接销毁产物，也能让下游停止逐步向上游传导。

---

## 17. OUTPUT 模式

OUTPUT 是强制清空模式。

行为：

```text
不再启动新节点
终止或收尾当前运行任务
输出所有运行资格物和缓存物料
输出完成后进入 STANDBY 或安装 pending 工序
```

提交新图触发的 OUTPUT 和螺丝刀 OUTPUT 可以共享底层输出逻辑，但语义不同：

```text
提交新图：允许继承主机和 NC 后，只输出不可继承内容
螺丝刀：不继承任何内容，全部输出
```

---

## 18. 排空模式边界记录

排空模式 `DRAINING` 暂不作为当前实现要求。

若未来实现，它的目标是：

```text
停止启动源头节点
允许只消费内部中间物的下游节点继续运行
尽量把旧图中间物加工成目标产物或副产物
```

但它存在以下边界：

```text
环图可能永不排空
下游节点可能需要外部辅料
副产链可能越排产生越多种物料
概率产物导致结果不确定
回收节点可能改变玩家期望的产物形式
能源不足时需要定义跳电语义
输出空间不足会阻塞排空
新图提交可能被旧图长时间阻塞
```

因此当前设计中，排空模式只记录为未来扩展，不影响现有提交和输出逻辑。

---

## 19. 跳电与电源关闭

电源关闭：

```text
停止运行调度
不启动新节点
已运行节点按当前机器语义处理
```

跳电：

```text
发生能量不足时停止机器
聊天栏提示坐标、机器名、节点或配方、耗时、耗能、并行和超频
不把已消耗但未完成的原料返回缓存
```

该语义用于模拟真实跳电损失。

## 21. 总结

v2 机器运行模型的核心是：

```text
图语义与运行状态分离
目标产物节点不是拓扑终点
调度器按物料边上游优先形成流水线
环由单循环物料和水位保护约束
提交新图可继承主机和 NC
螺丝刀输出仍然强制清空
排空模式暂只记录边界，不作为当前实现要求
```

该模型为后续改进调度顺序、复杂环支持、换图体验和性能优化提供统一语义基础。
