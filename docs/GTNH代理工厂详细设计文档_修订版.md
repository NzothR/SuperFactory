# GTNH 代理工厂详细设计文档（修订版）

## 1. 文档目的

本文档描述超级代理工厂的定位、结构模型、主机代理规则、输入输出模型、配方匹配流程，以及普通配方与特殊配方的代理执行规则。

超级代理工厂用于统一各种形状各异、结构复杂度不同，但本质上都暴露为某个 GT 配方表的多方块机器。玩家将对应多方块主机放入代理工厂后，代理工厂读取该主机暴露的配方表，并使用自身统一的输入舱、输出舱、能源舱和并行/超频参数执行对应配方。

它的核心价值是：

```text
把各种不同外形、不同仓室布局、不同结构体积的多方块机器，统一封装为一个标准化的代理执行入口。
```

玩家不再需要为每一种多方块机器单独搭建完整结构，而是通过提交对应主机，让代理工厂获得该机器的配方代理能力。

---

## 2. 核心定位

超级代理工厂是一台：

```text
多方块主机代理执行工厂
```

更具体地说，它是一种用于统一多方块结构差异的代理多方块：

```text
主机决定可代理的配方表
代理工厂提供统一的输入、输出、能源、并行和超频执行环境
```

它的作用是：

```text
读取主机槽中的 GT 多方块控制器
识别该控制器暴露的 RecipeMap 列表
缓存当前可代理配方表
允许玩家在多个 RecipeMap 之间切换代理模式
从自身输入舱室读取物品与流体输入
使用自身能源系统执行配方
按代理工厂参数计算并行、超频、耗时和耗能
将产物写入自身输出舱室
支持部分特殊多方块或特殊配方的适配路径
```

从玩家体验上看，代理工厂相当于：

```text
我提供某台多方块机器的主机
代理工厂负责用标准结构帮我运行这台机器能运行的配方
```

这样可以把复杂多方块的结构差异转化为主机差异，使自动化、批量运行和高并行处理更统一。

---

## 3. 结构模型

### 3.1 基础结构

代理工厂使用 3×3×3 空心外壳，结构与集成工厂初始外壳保持一致：

```text
3×3×3 空心立方体
正面中心为控制器
中心为空气
外壳可由机器外壳方块组成
允许输入总线、输入仓、输出总线、输出仓、维护仓、能源仓或异种能源仓替代外壳
```

该结构的目的不是复刻被代理多方块的原始结构，而是提供一个统一的代理执行平台。

### 3.2 主机槽

代理工厂拥有主机槽，用于放入被代理多方块的控制器。

主机槽中的控制器决定代理工厂当前可以访问哪些配方表。

机器读取主机槽后缓存：

```text
控制器 meta
控制器显示名
控制器堆叠数量
控制器暴露的 RecipeMap 列表
可用代理模式列表
```

其中：

```text
控制器类型决定可代理配方表
控制器堆叠数量映射代理工厂最大并行上限
RecipeMap 列表决定当前可切换的代理模式
```

### 3.3 主机堆叠与代理并行

主机槽允许同类控制器堆叠。

控制器堆叠数量用于表示代理工厂获得了多个同类多方块主机的代理能力，因此可以映射为更高的最大并行上限。

例如：

```text
主机槽中有 大型研磨塔主机 × 4
```

则代理工厂可以理解为拥有 4 台大型研磨塔的代理执行资源，并据此提高并行上限。

---

## 4. RecipeMap 代理模式

### 4.1 RecipeMap 缓存

主机槽中的控制器可能暴露一个或多个 RecipeMap。

代理工厂会缓存这些 RecipeMap：

```text
availableRecipeMaps = controller.getAvailableRecipeMaps()
```

如果某个主机只对应一个配方表，则代理工厂直接使用该配方表。

如果某个主机对应多个配方表，则玩家可以切换当前代理模式。

### 4.2 代理模式切换

代理模式可以通过以下方式切换：

```text
螺丝刀右键
```

切换内容为：

```text
currentRecipeMap
```

每次配方检查只在当前 `currentRecipeMap` 中进行。

### 4.3 配方表显示

GUI 应显示：

```text
当前主机名称
当前主机堆叠数量
当前代理 RecipeMap 名称
可切换 RecipeMap 列表
当前并行上限
当前超频与能源模式
```

---

## 5. 输入模型

代理工厂使用自身输入舱室读取配方输入，而不是读取被代理多方块原结构的仓室。

支持输入来源包括：

```text
普通输入总线 / 输入仓
ME 或兼容输入舱室
双输入 / 样板类输入舱室
输入隔离分组
```

### 5.1 普通输入路径

在未启用输入隔离时，代理工厂使用 GT 多方块基类的聚合输入视图读取物品和流体。

该路径能够兼容 ME 输入舱室，因为 ME 舱室可能不把真实库存暴露为普通 bus slot，但会通过多方块输入聚合 API 暴露给配方检查。

### 5.2 输入隔离路径

输入隔离启用时，机器按颜色和仓位分组读取输入。

输入隔离用于避免玩家显式隔离的输入被重新混合。

这适合以下场景：

```text
同一配方表中存在多个相近配方
多个输入组需要分别尝试
玩家希望锁定某组输入只服务于某类配方
```

### 5.3 双输入 / 样板类输入路径

双输入或样板类输入舱室拥有独立读取路径。

机器读取每个样板库存的：

```text
共享物品
槽位物品
槽位流体
```

并按样板槽独立尝试配方。

该路径适合高并行自动化和 AE 样板驱动输入。

---

## 6. 配方匹配流程

每次配方检查时，代理工厂执行以下流程：

```text
1. 检查主机槽是否存在有效控制器
2. 检查当前 RecipeMap 是否存在
3. 优先尝试双输入 / 样板类输入舱室
4. 未启用输入隔离时尝试机器聚合输入视图
5. 启用输入隔离时按输入组尝试
6. 遍历当前 RecipeMap 中的非 fake、启用配方
7. 对候选配方进行输入匹配、特殊物品校验、输出空间检查和能源检查
8. 生成执行计划
9. 消耗真实输入
10. 设置运行状态
```

查询视图与真实消耗视图分离：

```text
queryItems / queryFluids 用于配方匹配
liveItems / liveFluids 用于真实消耗
```

普通机器聚合输入视图需要通过机器 API 消耗输入，而不是直接修改查询数组中的栈数量。

这样可以兼容普通输入总线、ME 输入舱室和特殊输入舱室的不同内部实现。

---

## 7. 执行计划模型

配方匹配成功后，代理工厂生成执行计划：

```text
ProxyRecipePlan {
    recipe
    recipeMap
    inputItems
    inputFluids
    outputItems
    outputFluids
    actualParallel
    duration
    euPerTick
    totalEU
    batchMultiplier
}
```

执行计划用于描述本次运行的完整结果：

```text
消耗什么
消耗多少
运行多久
每 tick 消耗多少 EU
最终输出什么
输出多少
```

执行计划生成成功后，机器才会真实扣除输入。

## 8. 并行、超频与耗能

代理工厂使用自身参数计算配方并行、超频、耗时和耗能。

### 8.1 并行计算

```text
inputBound = 当前输入可支持的最大并行
parallelLimit = 控制器堆叠数量映射的最大并行
powerBound = 当前能源输入可支持的最大并行
actualParallel = min(inputBound, parallelLimit, powerBound)
```

其中：

```text
inputBound 由真实可消耗输入决定
parallelLimit 由主机堆叠数量和机器参数决定
powerBound 由能源仓、无线电网或异种能源输入决定
```

### 8.2 超频计算

代理工厂按 GT 风格完美超频计算耗时和 EU/t：

```text
baseDuration = 配方基础耗时
baseEUt = 配方 EU Input
ocResult = perfectOverclock(baseDuration, baseEUt, machineTier, extraOC)
duration = ocResult.duration
euPerTick = ocResult.euPerTick
```

普通配方在有线模式下默认允许自动完美超频；无线模式不自动超频，只应用玩家设置的手动超频。若普通配方的手动超频参数大于 0，则手动超频覆盖自动超频规则。

鸿蒙之眼配方无论有线或无线都不自动超频，只应用手动超频。

为避免无意义的能量浪费，手动超频在应用前会先计算“刚好压到 1 tick 所需的完美超频次数”。若玩家设置的手动超频次数超过该值，则实际应用该 1 tick 上限次数；GUI 中“当前应用OC次数”显示的始终是本次执行计划真实应用的超频次数。

### 8.3 批处理

批处理可以在输入充足且耗时较短时扩大一次执行的实际处理量，并相应拉长本次运行时间。

批处理的目标是减少高并行短耗时配方带来的 tick 级频繁启动开销。

### 8.4 无线模式

无线模式启用时，代理工厂在配方开始时一次性预扣总 EU：

```text
totalEU = euPerTick × duration
```

若配方启动失败，已预扣能量必须退回。

## 9. 普通配方代理

普通配方代理使用 GTRecipe 的常规字段：

```text
mInputs
mFluidInputs
mOutputs
mFluidOutputs
mChances
mDuration
mEUt
```

普通配方运行条件：

```text
存在可匹配输入，或该配方本身是合法的无消耗输入配方
配方必须是耗能配方，即有效 EU Input > 0
输出空间足够
当前能源输入或无线电网足够
当前 RecipeMap 与主机代理模式匹配
```

代理工厂不代理发电配方。对于普通 `GTRecipe`，若 `mEUt < 0`，该配方会被视为发电配方并跳过。

概率产物使用 GT 风格批量概率结算，而不是逐个物品随机，以避免后期大并行下的性能问题。

---

## 10. 回收机代理

回收机配方是特殊代理路径，不依赖普通 RecipeMap 的输出结算。

规则：

```text
输入为任意可被 GTModHandler.getRecyclerOutput 接受的物品
每个输入物品消耗 1 个
输出概率固定为 12.5%
使用批量概率算法结算废料
基础耗时为 45 tick
基础耗能为 1 EU/t
```

回收机代理用于把 IC2 回收机类行为纳入代理工厂统一输入、输出、能源和并行框架中。

---

## 11. 鸿蒙之眼代理

鸿蒙之眼配方不是普通 GTRecipe 字段的完整表达。其真实输入、输出和耗能需要从 `EyeOfHarmonyRecipe` 中读取。

代理工厂对鸿蒙之眼采用特殊有效值模型：

```text
有效流体输入：
  Hydrogen = eyeRecipe.getHydrogenRequirement()
  Helium   = eyeRecipe.getHeliumRequirement()

有效物品输出：
  eyeRecipe.getOutputItems()

有效流体输出：
  eyeRecipe.getOutputFluids()

有效耗时：
  eyeRecipe.getRecipeTimeInTicks()

有效 EU Input：
  max(eyeRecipe.getEUStartCost(), ceil(eyeRecipe.getEUOutput() / eyeRecipe.getRecipeEnergyEfficiency()))

有效 EU/t：
  ceil(有效 EU Input / 有效耗时)
```

鸿蒙之眼的 `EUOutput` 用于反推本次配方所需 EU Input。

鸿蒙之眼的 Hydrogen 与 Helium 输入仅支持通过存储输入仓（ME）供给。普通输入仓、四重输入仓等常规流体输入仓不作为鸿蒙之眼氢氦供给路径；混合残留流体时应以 ME 存储输入为准。

鸿蒙之眼的真实物品输出、流体输出和流体输入可能超过单个 `ItemStack` 或 `FluidStack` 的 `int` 数量上限。代理工厂应使用长整数批量拆分，将大数资源拆为多个普通栈或流体栈，再交给现有输出与消耗逻辑处理。

---

## 12. 输出模型

输出分为物品和流体：

```text
物品输出写入输出总线
流体输出写入输出仓
```

输出检查必须在消耗输入前完成。

若输出空间不足，本次执行计划不启动。

### 12.1 输出变换参数

代理工厂支持可选的输出变换参数：

```text
物品输出倍率
流体输出倍率
物品最小输出限制
物品最大输出限制
流体最小输出限制
流体最大输出限制
最小运行时间限制
最大运行时间限制
```

这些参数只作用于代理工厂最终执行计划，不修改原始配方。

### 12.2 大数输出拆分

当输出数量超过普通 `ItemStack` 或 `FluidStack` 上限时，代理工厂使用长整数统计总量，再拆分为多个合法栈或流体栈写入输出舱室。

---

## 13. 配方锁

配方锁开启后，机器在第一次成功匹配配方时建立 `SingleRecipeCheck`。

锁定内容记录真实消耗输入的前后状态。

查询用的非消耗标记、编程电路和特殊物品仍参与匹配校验，但不作为每次运行的消耗材料。

配方锁用于提升重复执行同一配方时的稳定性和性能。

---

## 14. 能源模型

代理工厂支持多种能源输入形式：

```text
普通能源仓
异种能源仓
无线电网
```

能源检查在执行计划生成阶段完成。

对于普通有线能源输入，机器检查当前能源输入能力是否足以支持本次运行的 `euPerTick`。

普通有线运行时，代理工厂的耗电行为与常规 GT 多方块机器执行配方一致：启动时设置负 `lEUt`，运行过程中由 GT 多方块基类按 tick 从能源仓或异种能源仓扣电。

对于无线模式，机器在启动时预扣：

```text
totalEU = euPerTick × duration
```

若启动过程未能完成，应退回预扣能量。

无线模式只用于消耗 EU，不用于接收或存入发电配方产出的 EU。代理工厂不会把任何配方作为发电任务写入无线电网。

## 17. 自动化接口

代理工厂可以暴露以下自动化能力：

```text
自动输入物品和流体
自动输出物品和流体
主机槽自动插入/取出，若设计允许
RecipeMap 模式切换，若设计允许
配方锁状态读取
运行状态读取
错误状态读取
```

对于 AE 或兼容输入舱室，代理工厂应优先使用多方块输入聚合 API 和对应的真实扣料接口，确保查询与真实扣料一致。

## 18. 设计原则总结

代理工厂的核心设计原则如下：

```text
1. 用统一结构代理各种形状不同但本质配方表相同的 GT 多方块。
2. 主机决定代理工厂当前可访问的 RecipeMap。
3. 主机堆叠数量映射代理工厂的最大并行能力。
4. 输入、输出、能源和维护由代理工厂自身结构统一提供。
5. 配方匹配使用查询视图，真实消耗使用 live 视图或机器 API。
6. 执行计划必须先完成输入、输出、能源检查，再真实扣料。
7. 普通配方使用 GTRecipe 字段代理执行。
8. 回收机、鸿蒙之眼等特殊配方使用独立适配路径。
9. 概率产物使用批量概率结算。
10. 大数物品与流体使用长整数统计并拆分输出。
11. 配方锁用于稳定重复执行同一配方。
12. 代理工厂的目标是统一多方块代理执行体验，提高高并行自动化便利性。
```

---

## 19. 最终简述

超级代理工厂是一台用于统一 GTNH 多方块结构差异的代理执行机器。玩家将某种 GT 多方块主机放入主机槽后，代理工厂读取该主机暴露的 RecipeMap，并使用自身统一的输入舱、输出舱、能源舱、并行和超频参数执行其中的配方。主机类型决定可代理的配方表，主机堆叠数量决定代理并行能力。代理工厂通过标准化结构封装不同外形、不同体积、不同仓室布局的多方块机器，使它们能够在同一套输入、输出和能源框架下运行，从而提升批量生产、AE 自动化和高并行加工的便利性。
